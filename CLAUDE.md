# CLAUDE.md

此文件为 Claude Code (claude.ai/code) 在此代码库中工作时提供指导。

# 语言与行为准则

  1. 始终使用**简体中文**回复，内部推理也必须使用简体中文，文档必须是中文。
  2. 编码前先思考——明确假设，存在歧义时列出多种方案，有更简单的方案时主动提出。
  3. 简单优先——用最少的代码解决问题。不添加未要求的功能、抽象或错误处理。
  4. 精准修改——只改动需要改的部分。不重构相邻代码，不改变格式。每行改动都必须能追溯到用户的请求。
  5. 目标驱动执行——定义可验证的成功标准。多步骤任务应先陈述计划再执行。


 # 行为准则

  行为准则，旨在减少常见的 LLM 编程错误。可根据项目需要与项目特定的指示说明合并使用。

  **权衡：** 这些准则偏向于谨慎而非速度。对于简单任务，请自行判断。

  ## 1. 编码前先思考

  **不要假设。不要隐藏困惑。要主动揭示并讨论不可避免的权衡。**

  在实现之前：

  - 明确陈述你的假设。如果不确定，请提问。
  - 如果存在多种解释，请呈现它们——不要默默选择。
  - 如果存在更简单的方案，请提出。当有充分理由时，要敢于反驳。
  - 如果有不清楚的地方，请停下来。说出令人困惑的地方，然后提问。

  ## 2. 简单优先

  **用最少的代码解决问题。不要画蛇添足。**

  - 不要添加需求之外的功能。
  - 不要为一次性使用的代码创建抽象。
  - 不要添加未请求的"灵活性"或"可配置性"。
  - 不要为不可能的场景添加错误处理。
  - 如果你能写 200 行但实际上 50 行就够，请重写。

  问自己："一位中级工程师会说这过于复杂吗？"如果是，请简化。

  ## 3. 精准修改

  **只触碰你需要改动的部分。只需清理你自己造成的混乱。**

  编辑现有代码时：

  - 不要"改善"相邻的代码、注释或格式。
  - 不要重构没有损坏的部分。
  - 保持与现有风格一致，即使你的风格有所不同。
  - 如果你注意到无关的死代码，请提及——但不要删除。

  当你的更改产生了孤立代码时：

  - 删除因你的更改而变得未使用的 import/变量/函数。
  - 不要删除预先存在的死代码，除非被要求。

  检验标准：每行更改都应该能直接追溯到用户的请求。

  ## 4. 目标驱动执行

  **定义成功标准。循环验证直到达成。**

  将任务转化为可验证的目标：

  - "添加验证" → "为无效输入（用户输入了任何不符合你程序预期或规则的内容）编写测试，然后让它们通过"
  - "修复 bug" → "编写测试来复现它，然后让测试通过"
  - "重构 X" → "确保测试在重构前后都能通过"

  对于多步骤任务，请陈述一个简要计划：

  ```
  1. [步骤] → 验证：[检查项]
  2. [步骤] → 验证：[检查项]
  3. [步骤] → 验证：[检查项]
  ```

  强有力的成功标准让你能够独立循环。弱标准（"让它能工作"）需要不断的澄清。

  ---

  **如果看到以下情况，说明这些准则正在起作用：** diff 中不必要的更改更少、由于过度复杂而导致的返工更少、澄清性问题出现在实现之前而非错误之后。
